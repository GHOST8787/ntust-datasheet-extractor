# 第一章　方法說明（對應四個任務）

## 1.1 任務一：用 VLM 解析尺寸圖面

封裝本體尺寸（長、寬、高）在多數 datasheet 裡只畫在 mechanical drawing 工程圖面，純文字與表格 parser 抓不到——這是傳統 OCR pipeline 的死角，也是上一版（V2，純文字 100 格對 76 格）的主要失分來源。

本版改用 VLM（Vision-Language Model）直接看圖：用 PyMuPDF 把 PDF 頁面光柵化成高解析 PNG（`Matrix(4.5x)`），餵給多模態模型 GPT-4o 判讀尺寸圖。效果反映在版本演進上——從 V2（純文字 100 格對 76 格）到 V12（多模態看圖 100 格對 85 格）一步多對約 9 格，尺寸欄位的改善是主因（例如 MSB30M 的長寬在純文字版被搞反，多模態看圖後修正為正確的 8.6 / 6.7）。

最終整合版在 V12 基礎上再疊加：(1) **V23 三模型投票**——三個獨立的視覺/抽取模型（GPT-4o 多模態 / Llama-4 vision / GPT-4o + TOC variant）對同一份 datasheet 各跑一次，逐欄取多數值；(2) **通用裁切尺寸表**——在 PDF 內搜尋「Package Outline / Mechanical Drawing / Mechanical Dimensions」等通用 anchor 文字，以其 bbox 往外擴展裁切尺寸表區塊，以高解析重新光柵化後再餵 VLM（聚焦注意力，移除 Land Pattern / Tape & Reel 等雜訊）；(3) **通用物理規則**——長寬取「含引腳的最外緣 overall span」（JEDEC 封裝慣例）、每個值取規格表的 Max 欄、尺寸寫成「中心值 ± 公差」時 Max = 中心值 + 公差。三個步驟全部對任何 datasheet 都成立，不依賴特定零件。

## 1.2 任務二：強制 100% 合法 JSON（不寫在 prompt 裡）

要讓模型穩定輸出可直接入庫的 JSON，靠的是 **API 解碼層的 JSON mode**，而不是在 prompt 裡拜託模型「請輸出 JSON」：

- Azure / OpenAI 相容後端：呼叫時帶 `response_format={"type": "json_object"}`
- 本地 Ollama：請求帶 `format="json"`

這兩者都在模型「產生 token 的階段」就用 logits 約束輸出必為合法 JSON，從根本上杜絕「模型多講一句廢話、包了 markdown code fence、JSON 中途截斷」等問題。實作位置：`llm_backend.py` 的 `AzureFoundryV1Backend.call / call_multimodal`（預設 `json_object`）與 `OllamaBackend`（帶 `format:json`）。作為第二道保險，另有寬鬆解析（去除 code fence、抓第一個 `{` 到最後一個 `}`），但主要保證來自 JSON mode 本身。

## 1.3 任務三：批次處理 10 筆、每欄 10 格對 5 格以上

10 份 datasheet 全部批次跑完，統一以嚴格容差評分：數值欄位 ±5% 容差、文字欄位（V_F / I_R）以全形頓號拆 token 後集合命中率 ≥ 50% 視為正確。結果見第三章。

## 1.4 任務四：本機 webapp

`app.py`（Streamlit），本機 `streamlit run app.py` 即可部署。兩個區塊：

- **區塊一「10 顆 SPEC 示範成果」**：讀已跑好的整合版結果，秒出、零 API 呼叫，逐格對照標答（紅底標示錯格），可收合。
- **區塊二「上傳 PDF 即時跑」**：可一次拖多檔，當場端到端抽取（GPT-4o 看圖 + 自我檢查）。此為單模型即時版（無多模型投票與裁切尺寸後處理，準確率低於完整整合版），非完整整合版；完整整合版是多層通用機制疊加，現場重跑成本高，故即時版用單模型展示流程。

# 第四章　為什麼這 91 格的成績「不是背答案」（零 overfit 論證）

高準確率最常被質疑的就是「會不會只是把這 10 顆的答案背起來」。本章從系統的決策機制本身，逐項說明整合版（91/100）為什麼不依賴特定答案。

## 4.1 先講限制：本報告不依賴第二組獨立泛化測試

嚴格驗證 overfit 的標準做法是「拿從未見過、且有可信標答的第二組資料」測泛化。本專案開發過程中雖曾蒐集 5 份額外 PDF 做粗略 holdout，但那組資料的標答由作者自行標註、未經正式核對，**不具備作業官方答案的可信度**。在「不希望任何非事實」的硬約束下，**本報告不將該組 holdout 數字作為證據**（這 5 份的相關內容與結果也不在繳交範圍，詳見 5.3）。以下論證**完全不依賴第二組資料**，只看 train 的官方比對結果與系統本身的程式邏輯。

## 4.2 整合版的所有機制都是通用機制

整合版（91/100）的方法由三個部分組成，每一個都不依賴特定答案：

1. **三模型投票（V23）**：用三個獨立的視覺/抽取模型對同一份 datasheet 各跑一次，逐欄取多數值。投票本身是「多模型共識」的通用機制，與是哪份 datasheet 無關。
2. **通用裁切尺寸表**：在 PDF 內搜尋「Package Outline」「Mechanical Drawing」「Mechanical Dimensions」等通用 anchor 文字，以其 bbox 往外擴展裁切尺寸表區塊，再以高解析光柵化餵 VLM。這是純 generic 的 PDF 結構處理，anchor 字串是工業界 datasheet 標準用語。
3. **通用物理規則**：所有實驗組共用同一套規則——長寬取「含引腳的最外緣 overall span」（JEDEC 封裝慣例）、每個值取規格表的 Max 欄（作業欄位名標的就是「Maximum」）、尺寸寫成「中心值 ± 公差」時 Max = 中心值 + 公差（標準計算）。這些規則只描述「怎麼正確地看圖」，不直接設任何數值、不點名任何零件。

光是「多模態看圖 + 三模型投票」這兩個與答案無關的通用機制，就已達到 100 格對 86 格（V23）。再加上通用裁切尺寸表 + 通用物理規則，把整合版推到 100 格對 91 格。這 91 格的提升完全建立在可泛化的通用機制上。

## 4.3 程式碼可驗證：系統沒有「逐格背答案」的途徑

不看任何分數，光讀整合版的程式碼（`_integrate_final.py` 與相關 extractor），也能確認系統不是在背這 10 顆的答案：

- **沒有寫死任何零件名稱**：所有處理邏輯找不到 `if part == "BAV99W"` 這類判斷，對所有零件一視同仁。
- **觸發用通用條件**：規則的觸發是「比例」或「文字模式」（anchor 字串），不是綁定特定零件的絕對值，因此具尺度無關的通用性。
- **同一套程式處理所有資料**：沒有針對個別零件的特例分支。
- **決策可複現**：整條流程是確定性函式（temperature=0），給定同樣輸入永遠得同樣輸出，沒有藏起來的查表。

## 4.4 對照 V37–V47：為什麼那些不算「不 overfit」的成績

開發過程中，V37–V47 系列的後處理規則確實把 train 分數推得更高（V40 100 格對 96 格、V44 / V47 100 格對 95 格），但 `OVERFIT_AUDIT.md`（2026-05-28 audit）認定這些版本對評分集 overfit，原因如下：

- **V37** 的「L = max(D, E) swap rule」來自「這 10 份 spec 都觀察到 L ≥ W」的統計，是直接擬合評分集；
- **V40** 的 `H_E ratio 1.4 ~ 1.8` 窗口上下界，是看著 train 那幾顆（BAS16 1.79 / BAV99W 1.76）的實際比例調出來的；
- **V44** 的「grep `Tamb` literal」是為了精準區分 BAV99W vs BAS16 的 Tj 計法 schema，這個區分本身是從 train 兩顆的 GT 反推；
- **V47** 在 V44 之上加 Llama swap 偵測，整體仍繼承 V40 / V44 的 train 調參（且 swap 機制在作業 10 份未觸發，結果與 V44 完全相同）。

這些規則的觸發條件**形式上**是 generic（比例、文字模式），但**實質上**門檻是依 train 那幾顆反推設定，不能保證在新 datasheet 上仍最佳。因此最終成果改採完全不含 part-specific 後處理的整合版（91/100），V37–V47 僅保留為過程記錄。詳細逐版判定見 `OVERFIT_AUDIT.md`。

## 5.2 三大發現

1. **VLM 看圖是這次躍升的關鍵**：從 V2（純文字 100 對 76）到 V12（多模態 100 對 85）一步多對約 9 格，主因是讓模型直接看 mechanical drawing 圖面，補上純文字 OCR 的死角。
2. **模型升級的效益遠大於 PDF 解析調校**：早期 V2 實驗的雙軸比較顯示，換模型造成的準確率落差（軸極差約 14.6%）遠大於換 PDF 解析方法的落差（約 3.6%），模型選擇是主要槓桿。（「軸極差」指固定一軸、變動另一軸時準確率的最大值減最小值，是比較兩個因子影響力的相對指標，非單一版本的格數準確率。）
3. **multi-product datasheet 是模型容量試金石**：MBR15U150 一份含多個型號變體，是僅存錯格的集中地之一。

# 第六章　尺寸抽取方法的學術對照實驗（四種主流路線 × 我們的實驗）

尺寸三欄（Maximum Length / Width / Height）是全部 10 個（從 PDF 抽取的）欄位中最難的，也是各版本失分最集中的地方——因為這三個值多半只畫在 mechanical drawing 工程圖面，且同一張圖常同時有「本體尺寸」與「含引腳外形」兩組符號，視覺模型很容易抽錯軸或抽到本體而非外形。

學術界在提升 VLM 結構化抽取準確率上，目前有四條主流路線：(一) Fine-tune 專門模型、(二) CV + VLM 區域裁切、(三) OCR + LLM 兩階段、(四) 結構化分步提示 + self-consistency。本章把這四條路線各當成一組獨立實驗，沿用第二版（V2）實驗報告的「控制變因 / 自變數 / 對照 baseline / 結果 / 成因分析」科學寫法，逐組記錄我們做了什麼、得到什麼確切結果、為什麼成功或退步。

## 6.1 實驗設計（共用設定）

本章四組實驗共用同一套控制變因，只變動「尺寸抽取方法」這一個自變數，藉此乾淨分離各方法的效果。

### 6.1.1 控制變因

| 控制項 | 值 |
| --- | --- |
| 任務 | 從 10 份 datasheet 抽尺寸三欄 L / W / H（共 30 格） |
| 視覺模型 | GPT-4o（Azure Foundry V1，multimodal） |
| Ground Truth | `specbook.xlsx`（外部給定標答） |
| 比對邏輯 | 數值 ±5% 容差（`reevaluate.cmp_numeric`，與 `evaluate_all.py` 一致） |
| Temperature | 0（確定性輸出；唯 self-consistency 組刻意多次取樣） |
| 物理規則 | 零 overfit 的通用領域定義（見 6.1.3），不點名任何零件 |

### 6.1.2 自變數：尺寸抽取方法

| 組別 | 方法 | 對應腳本 |
| --- | --- | --- |
| 對照組 | 三模型投票（V23）對尺寸三欄的結果 | `output/_archive/v23_majority_vote_3model/` |
| 路線一 | Fine-tune 專門模型 | 未做（判定不適用，見 6.2） |
| 路線二 | CV + VLM 區域裁切 | `_exp_dim_crop.py`、`_exp_dim_crop2.py`、`crop_extractor.py` |
| 路線三 | OCR + LLM 兩階段 | `_exp_2stage.py` |
| 路線四 | 結構化分步提示 + self-consistency | `_exp_dim_v2.py` |
| 單步基準 | 整頁高清 + 單步物理規則（裁切前的對照） | `_exp_overall.py` |

### 6.1.3 零 overfit 的通用物理規則

所有實驗組共用同一套**通用領域定義**，只描述「怎麼正確地看圖」，不直接設任何數值、不點名任何零件、不照這 10 份的答案反推：

1. **長、寬取「含引腳的最外緣 overall span」，不取本體尺寸**——例如 SOT-23 的 E（lead span）是含引腳外形、E1（body）是本體，Width 取含引腳那個較大值。
2. **每個值取規格表的 Max 欄**（最右欄），不取 Min / Nom / Typ。
3. **尺寸寫成「中心值 ± 公差」時，Max = 中心值 + 公差**（如 1.55 ± 0.1 → 1.65）。

這套規則有 datasheet 與 JEDEC 封裝慣例佐證，屬於通用領域知識而非答案擬合：DFLS160 的標答取含引腳的 E = 3.9 而非 body E1 = 3.0；CD4148WTP 標示 1.55 ± 0.1，依規則 Max = 1.65，與標答一致。這正是規則「可泛化到任何 datasheet」而非「背這 10 顆」的證據。

### 6.1.4 對照組基準

對照組為 **V23 三模型投票** 在尺寸三欄的表現：**17 / 30**。V23 是本專案最乾淨、任何尺度都站得住的版本（純通用手法，無 part-specific 規則）。本章所有方法都以「能否在不引入 overfit 的前提下，把這 30 格從 17 格往上推」為衡量標準。

另設一個**單步基準**：`_exp_overall.py` 用整頁高清圖 + 6.1.3 的單步物理規則，尺寸命中約 **21 / 30**（VLM 對同一張圖每次讀數有隨機差異，重跑時此基準在約 17~21 / 30 間浮動）。路線二的裁切實驗即以此單步基準為起點，只改「裁切 vs 整頁」一個變數。

## 6.2 路線一　Fine-tune 專門模型（未做，判定不適用）

**方法。** 學術上最直接的路線是收集大量「datasheet 圖面 → 尺寸標註」配對，對視覺模型做監督式微調（或 LoRA），讓模型把「含引腳外形 vs 本體」「Max 欄取值」內化成權重。

**為什麼判定不適用（而非沒做）。** 本作業評分集只有 10 顆元件、共 30 個尺寸格。Fine-tune 需要的標註資料量遠超此規模（學術慣例至少數百到數千筆），且需要 GPU 訓練資源。在只有 10 顆的情況下硬做微調有兩個致命問題：一是資料量根本不足以學到可泛化的特徵；二是**用這 10 顆的答案去訓練，等於把評分集本身餵進模型權重，這正是最嚴重的 overfit**。因此在零 overfit 的硬約束下，這條路線從設計上就被排除，不是「做不出來」，而是「做了就違規」。

**結論。** Fine-tune 在本任務規模與零 overfit 約束下不適用；正確做法是把力氣放在「不依賴特定答案的通用抽取機制」（路線二）。

---

## 6.3 路線二　CV + VLM 區域裁切（成功路線）

**方法。** 不把整頁丟給 VLM，而是先用電腦視覺 / PDF 結構定位尺寸表所在區塊，裁切出來高解析重新光柵化，再餵給 VLM。`crop_extractor.py` 的做法是：在 PDF 內搜尋「Package Outline / Mechanical Drawing / Mechanical Dimensions」等通用 anchor 文字，以其 bbox 往外擴展（上 50、下 400、左 50、右 500 points）框出表格區，以 zoom = 6.0 裁切成高清 PNG。這是純 generic 的 PDF 結構處理，不依賴任何 train set 內容。

**控制變因。** 與單步基準（`_exp_overall.py`，整頁 30 對 21）**只差「裁切 vs 整頁」一個變數**，prompt 完全沿用 6.1.3 的單步物理規則，單次呼叫、裁切失敗的顆 fallback 整頁。

**結果（兩次遞進）。**

| 子實驗 | 改動 | 尺寸命中 |
| --- | --- | --- |
| 對照（V23 投票） | — | 17 / 30 |
| 單步基準（整頁 + 單步規則） | `_exp_overall.py` | 21 / 30 |
| 路線二之一 | `_exp_dim_crop.py`：裁切尺寸表 + 單步規則 | **23 / 30** |
| 路線二之二 | `_exp_dim_crop2.py`：再補通用標題關鍵字 `DIMENSIONS`（僅對裁切失敗的顆）+ ± 公差規則 + Height 符號通用化（A / T / H） | **24 / 30** |

**為什麼成功。** 裁切等於替 VLM 做了「視覺注意力聚焦」——把整頁的 Land Pattern、Tape & Reel、電氣表等雜訊先移除，模型只看尺寸表，少抽錯軸。這與 V2 報告觀察到的「camelot 抽出近兩倍字數反而稀釋 attention 退步」是同一條原理的正面版：**減少輸入雜訊比增加輸入資訊更有效**。第二步的兩個補強（`DIMENSIONS` 標題、± 公差換算）都是通用規則，且只在原本裁切失敗的顆才啟用，保住已成功的戰果、不動到已對的顆，所以能再 +1 而無副作用。整條路線從頭到尾沒有點名任何零件、沒有寫死任何數值，是乾淨可辯護的提升。

**整合驗證。** `_integrate_final.py` 把這套裁切尺寸抽取整合回 V23 三模型投票、跑完整 10 顆 × 10 欄，整體達 **100 格對 91 格**（其中尺寸這次為 22 / 30）。尺寸這次落在 22 而非 24，正是 VLM 對同一張圖每次讀數的隨機差異，並非邏輯退步——這也再次印證 6.1.4 提到的 run-to-run 變異是 VLM 的固有性質。整合後的 100 格對 91 格 完全建立在通用機制（V23 投票 + 通用裁切 + 通用物理規則）之上，不含任何 part-specific 後處理。

## 6.4 路線三　OCR + LLM 兩階段（退步）

**方法。** 學術上常見的「先抽文字、再分析」路線：階段一讓 GPT-4o 把 Package Outline 表忠實 OCR 成結構化 JSON（每個符號的 min / nom / max），不做任何判讀；階段二改用純文字 LLM 讀那份 JSON，套硬性符號對應規則抽 L / W / H。`_exp_2stage.py` 的階段二規則是：**D → Length、E → Width、A → Height**（硬性對應，不論數值大小、不交換，且只准用 A / D / E 三個主符號，禁用 D1 / E1 / H_E 等變體）。

**對照組基準。** V23 投票 17 / 30；本實驗測 V23 尺寸錯的 7 顆共 21 格。

**結果。** **4 / 21**，大幅退步。

**為什麼退步。** 核心敗因是**階段二的硬性軸對應太死**。它假設「D 必是長、E 必是寬、A 必是高」，但 datasheet 的符號與軸的對應其實依封裝而異：

- DFLS160（SMA 類）的 E 其實是含引腳全長、E1 才是本體長，硬把 E 當 Width 直接抽錯（log：L 抽到 1.78 / 標答 3.9、W 抽到 3.7 / 標答 1.93，整個對調）。
- 規則又禁用 D1 / E1 / H_E 等變體符號，但很多封裝的「含引腳外形」恰恰標在這些變體上，於是正確答案在階段二根本不在可選範圍內（CD4148WTP 三欄全抽不到、回 null）。

換句話說，OCR 階段雖然忠實，但把「判讀」推遲到一個只看文字、看不到圖的純文字 LLM，再用一套過度簡化的硬規則去套，等於丟掉了 VLM「看圖判斷哪個符號量哪個軸」的最大優勢。這條路線把原本 VLM 直覺能做對的事，硬拆成「OCR + 死規則」反而做壞。

---

## 6.5 路線四　結構化分步提示 + self-consistency（退步）

**方法。** 不拆成兩個模型，而是在同一次 multimodal 呼叫內強制 VLM 兩步走：步驟一先忠實列出表中「每一個」符號的軸（長軸 / 短軸 / 高度）、是含引腳還是本體、min / nom / max；步驟二才依「含引腳最大外形 + 取 Max 欄」物理規則對應 L / W / H。再加 **self-consistency**：每顆跑 3 次，逐欄取多數值。`_exp_dim_v2.py` 即此設計。

**控制變因。** 同樣餵整頁高清圖、同樣的通用物理規則，與單步基準（整頁 30 對 21）只差「強制分步列表 + 跑 3 次取多數」這個變數。

**對照組基準。** V23 投票 30 對 17、單步整頁基準 30 對 21。

**結果。** **16 / 30**，比單步基準退步約 5 格，甚至低於 V23 對照組。

**為什麼退步。** 兩個原因疊加：

1. **強制分步打亂了 VLM 的直覺。** 逼模型先把每個符號逐一歸類（軸 / 含引腳 vs 本體），等於要它在「還沒整體看懂圖」前就先做局部標註；一旦某個符號被歸錯類（例如把含引腳的 E 標成「本體」），步驟二就照錯誤前提去對應，錯誤被前一步鎖死。VLM 本來「整張圖一起看、憑視覺直覺判斷哪個是外形」反而更準——單步基準的 30 對 21 就是證據。

2. **self-consistency 固化了系統性錯誤。** 取多數投票只能消除「隨機」雜訊，對「系統性」偏誤無能為力。如果 VLM 對某張圖每次都傾向抽到同一個錯誤符號（這是系統性偏誤，不是隨機），跑 3 次的多數仍是那個錯誤值，投票反而把偶然抽對的那一次蓋掉。換言之，self-consistency 在這裡把「穩定地錯」當成「正確」加以固化。

這兩點合起來說明：對已經有不錯視覺直覺的 VLM，**過度結構化的分步指令是負作用**；真正有效的是改善「給模型看什麼」（路線二的裁切），而不是「強迫模型怎麼想」。

## 6.6 四組對照總表與結論

| 路線 | 方法 | 對照基準 | 結果 | 判定 |
| --- | --- | --- | --- | --- |
| 一 | Fine-tune 專門模型 | — | 未做 | 不適用（10 顆不足 + 需 GPU + 用評分集訓練即 overfit） |
| 二 | CV + VLM 區域裁切 | V23 30 對 17 / 單步 30 對 21 | 裁切 30 對 23 → 加通用補強 30 對 24 | 成功（整合回 100 格達 100 格對 91 格） |
| 三 | OCR + LLM 兩階段 | V23 30 對 17 | 4 / 21 | 退步 |
| 四 | 結構化分步 + self-consistency | 單步 30 對 21 | 16 / 30 | 退步 |

**四組各一句結論：**

1. **Fine-tune**：在 10 顆規模與零 overfit 約束下不適用，硬做等於把評分集餵進權重。
2. **CV + VLM 裁切（成功）**：替 VLM 移除整頁雜訊、聚焦尺寸表，30 對 23 → 30 對 24，整合回完整 100 格達 100 格對 91 格，全程通用無 overfit。
3. **OCR + LLM（退步）**：硬性軸對應太死、又禁用變體符號，丟掉 VLM 看圖優勢，退到 4 / 21。
4. **結構化分步 + self-consistency（退步）**：強制分步打亂 VLM 直覺、self-consistency 固化系統性錯誤，退到 16 / 30。

**整體啟示。** 提升 VLM 尺寸抽取，有效的是改善「輸入品質」（裁切聚焦），無效甚至有害的是「強加流程約束」（兩階段硬規則、強制分步）。這與第五章「VLM 看圖是躍升關鍵」「減少雜訊勝過增加資訊」的發現一致，且這條成功路線完全建立在可泛化的通用機制上，符合本作業零 overfit 的硬約束。
